# ADR 002: Database Baseline

## Status

Accepted

## Context

Проекту нужна стартовая схема PostgreSQL, которая с самого начала поддерживает:

- auth и роли для web UI
- каталог мест и линии
- назначение мест на дату
- историю изменений и перемещений
- импорт и карту этажей
- строгие ограничения на уровне БД

## Decision

В baseline-миграции принимаем такие принципы:

- первичные ключи на `uuid`
- audit-friendly таблицы с `created_at` и `updated_at`
- enum-ы для ключевых доменных состояний
- partial unique indexes для активных заявок и активных назначений
- `daterange` и exclusion constraints для постоянных закреплений
- отдельные таблицы `reservation_events` и `parking_movements` для полной трассировки
- отдельные таблицы `parking_place_maps` и `parking_place_map_zones` для карты парковки

## Rationale

Почему не делаем минималистичную плоскую схему:

- у проекта много правил по датам, статусам и конфликтам
- история назначений является обязательной частью продукта
- ручные и автоматические изменения нужно различать на уровне данных
- карта этажей и кликабельные зоны требуют явной модели, а не JSON в одной колонке

## Consequences

Плюсы:

- сильные ограничения на уровне БД
- меньше шансов на гонки и неконсистентные назначения
- проще строить audit и историю без последующего painful refactor

Минусы:

- схема получается крупнее уже на старте
- часть логики требует аккуратной миграционной дисциплины
