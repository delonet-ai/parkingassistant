# Parking Assistant

Система управления офисной парковкой с `backend API`, `admin web UI` и `bot adapter` для `Yandex Messenger`.

Проект нужен для управления:

- постоянными парковочными местами
- временной отдачей закрепленных мест
- заявками сотрудников без постоянного места
- гостевыми парковками
- multi-линиями на 2-3 машины
- очередью на место
- приоритетными ручными назначениями администратора
- доступом к контактам тех, кто блокирует выезд

## Цели

Система должна:

- хранить всю бизнес-логику только в backend
- предоставлять API для бота и web UI
- учитывать правила доступности мест до `19:00` и после `19:00`
- поддерживать резерв минимум `5` мест для гостей
- учитывать ранние выезды и конфликты в multi-линиях
- вести audit log по важным действиям
- выполнять фоновые правила на `19:00`, `07:00` и в начале дня

## Основные роли

`Администратор`

- управляет парковкой через web UI
- назначает места сотрудникам и гостям
- управляет очередью
- просматривает парковку, конфликты и журнал действий

`Сотрудник с постоянным местом`

- отдает свое место на день или диапазон
- отменяет будущие отдачи
- указывает плановое время выезда на завтра для multi-мест
- фиксирует фактическую позицию в линии

`Сотрудник без постоянного места`

- запрашивает место на 1 день
- попадает в очередь
- указывает время выезда на завтра для multi-места
- фиксирует позицию в линии
- получает контакты тех, кто стоит впереди

`Гость`

- создается только администратором
- отдельного интерфейса не имеет

## Архитектура

Состав системы:

- `backend API`
- `admin web UI`
- `bot adapter` для `Yandex Messenger`
- `PostgreSQL`
- `scheduler/background jobs`

Принципиальные ограничения:

- backend stateless
- PostgreSQL как основная БД
- bot и admin UI работают только через backend API
- единая timezone для всех правил по датам и времени
- ошибки должны быть понятны и для бота, и для админки

## Бизнес-правила

### Закрепленные места

- по умолчанию закрепленное место занято владельцем
- владелец может отдать место на день или диапазон
- будущие отдачи можно отменять
- если после `19:00` следующий день уже зафиксирован в распределении, вернуть следующий день нельзя

### Освобожденные места

- освобожденное `single`-место в первую очередь предназначено для гостей
- для сотрудников такие места открываются только после `19:00` накануне
- администратор может назначить сотрудника вручную в любой момент

### Сотрудники без места

- самостоятельно можно запросить место только на 1 день
- до `19:00` доступны только неприоритетные multi-места
- после `19:00` доступны все разрешенные места, если это не нарушает гостевой резерв
- резерв гостевых мест нельзя снижать ниже `5`

### Очередь

- очередь обрабатывается один раз в начале дня
- назначение идет по порядку очереди
- ручное назначение администратора имеет приоритет
- вручную назначенные пользователи исключаются из очереди на этот день

### Гости

- гость создается только администратором
- у гостя обязательно есть приглашающий сотрудник
- гость получает место сразу, без очереди
- приоритет выдачи гостю: `single` -> `double` -> `triple`

### Multi-линии

- для линий 2/3 мест хранится группа линии и фактическая позиция
- позиции: `1`, `2`, `3`
- пользователь сам фиксирует свою позицию через бот
- система должна уметь определить цепочку "кто перед кем"

### Время выезда

- указывается только на завтра
- редактирование допустимо только до `07:00` дня выезда
- значение имеет смысл только для выезда раньше `18:00`
- если кто-то может перекрыть ранний выезд, система должна показать предупреждение

### Контакты и конфликты

- можно запросить контакты тех, кто стоит впереди
- если впереди сотрудник, показываются его контакты
- если впереди гость, нужно направлять пользователя к администратору
- запросы контактов логируются
- при назначении гостя перед сотрудником с ранним выездом администратор получает предупреждение

## Ограничения данных

Нужны как минимум такие ограничения:

- одно место не может быть назначено двум сущностям на одну дату
- один пользователь не может иметь более одной активной заявки на одну дату
- одна позиция в линии не может быть занята двумя пользователями на одну дату

Транзакции обязательны для:

- назначения места
- постановки в очередь
- ручного назначения
- фиксации позиции в линии
- отмены и изменения отдачи места

## Фоновые задачи

После `19:00`:

- зафиксировать следующий день для уже отданных мест
- раскрыть доступность мест сотрудникам по правилам
- проверить резерв гостевых мест

После `07:00`:

- заблокировать редактирование времени выезда на текущий день

В начале дня:

- обработать очередь
- назначить места из доступного пула
- учесть ручные назначения администратора
